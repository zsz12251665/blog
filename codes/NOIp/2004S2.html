Solution

Well, let's define that the bunches of fruits are #fruit#(_1_) to #fruit#(_#n#_). And for every #i# in [1,#n#-1], #fruit#(_#i#_) is smaller than #fruit#(_#i#+1_). When we need to combine two bunches of fruits, choose #fruit#(_1_) and #fruit#(_2_) can keep the least cost since they are the smallest ones. And then we add the new bunch which is (#fruit#(_1_)+#fruit#(_2_)) back to the fruit array. Now we have (#n#-1) bunches now. Repeat these steps, we can finally have a single bunch with least cost.

What's important is that we can use a heap to keep the fruit array. The heap is a data structure with an O(lg(#n#)) insert and an O(lg(#n#)) extract. With the heap to keep the #fruit# array, we can reduce the time complexity to O(#n#lg(#n#)), which is fast enough to solve the problem.

While building up the heap, I found a way to make fewer mistakes when extracting. That is fill the null part with maxlongint. That means if a point has no child, it has a child or two children with maxlongint. That can be very helpful because I don't need to judge if the point has child or not by the size of the heap.
